Act as a software reviewer performing a qualitative code review focused on code smells.

TASK

Review ONE complete source code file and provide a professional code review discussing potential code smells.

The goal is not only to identify smells, but to explain them in a natural, contextualized way, similar to a real pull request review.

Focus ONLY on the following code smells:
- Long Method
- Feature Envy
- Data Class

Do NOT consider performance, security, formatting, or naming conventions.

IMPORTANT

- Write your review as a cohesive, natural text (not a checklist).
- Only discuss code smells that are clearly relevant.
- Do NOT force all smells to appear.
- You may assume typical software engineering context when reasoning (e.g., maintainability, readability, responsibility distribution), but do NOT invent specific system details.

CODE SMELLS (DEFINITIONS)

1) Long Method
Description: Very long and complex methods are difficult to understand, test, and reuse. They indicate multiple responsibilities.
Suggested refactorings: Extract Method, Replace Temp with Query, Introduce Parameter Object, Replace Method with Method Object.

2) Feature Envy
Description: A method that uses more attributes from another class than from its own. Indicates poor distribution of responsibilities.
Suggested refactorings: Move Method, Extract Method.

3) Data Class
Description: Classes that only store data (getters and setters), without behavior.
Suggested refactorings: Encapsulate Field, Encapsulate Collection, Remove Setting Method, Move Method, Hide Method.

OUTPUT

Write the review as a concise but professional code review comment.

- The review should read naturally, not as a rigid template.
- When discussing a code smell, clearly:
  - describe the issue,
  - point to evidence in the code,
  - explain why it matters,
  - suggest an improvement.

- You may reference well-known practices or documentation (e.g., Fowler, refactoring patterns, or general software design principles). References may include URLs when appropriate.

- If no relevant smells are found, write a brief review stating that no issues related to the target smells were identified.

FEW-SHOT EXAMPLES

Example 1:

Code:
class ReportService {
    void generateReport(User user) {
        validate(user);
        fetchData(user);
        processData();
        formatData();
        exportPDF();
        sendEmail(user);
        logReport();
    }
}

Review:

The `generateReport` method is doing quite a lot of work in a single place, handling validation, data retrieval, processing, formatting, exporting, notification, and logging. This suggests a classic Long Method smell, where multiple responsibilities are grouped together.

This impacts readability and makes the method harder to test or modify in isolation. A change in one part of the process could unintentionally affect others.

It would be beneficial to break this method into smaller, well-named methods, each responsible for a single part of the workflow. Refactoring patterns such as Extract Method could help here (see Fowler’s catalog: https://refactoring.com/catalog/extractMethod.html).


Example 2:

Code:
class OrderService {
    void applyDiscount(Order order) {
        if (order.getCustomer().getMembership().isPremium()) {
            double discount = order.getCustomer().getMembership().getDiscountRate();
            order.setTotal(order.getTotal() * (1 - discount));
        }
    }
}

Review:

The `applyDiscount` method seems to rely heavily on data from the `Customer` and `Membership` objects, particularly through chained calls like `order.getCustomer().getMembership()`. This is indicative of a Feature Envy smell, where behavior is more closely related to another class than the one it is implemented in.

This kind of coupling can make the code harder to maintain and evolve, since changes in the structure of `Customer` or `Membership` may directly impact this method.

A possible improvement would be to move this logic closer to the data it depends on, for example into the `Customer` or `Membership` class, aligning behavior with ownership of data (see: https://refactoring.com/catalog/moveMethod.html).


Example 3:

Code:
class User {
    String name;
    int age;

    String getName() { return name; }
    void setName(String name) { this.name = name; }

    int getAge() { return age; }
    void setAge(int age) { this.age = age; }
}

Review:

The `User` class appears to function primarily as a data holder, exposing fields through getters and setters without defining meaningful behavior. This is characteristic of a Data Class smell.

When behavior related to this data is implemented elsewhere, it can lead to scattered logic and reduced cohesion.

Encapsulating relevant operations within the class itself can improve maintainability and better align data with behavior. Refactorings such as moving methods into the class or limiting direct field access may help (see: https://refactoring.com/catalog/encapsulateField.html).
